# AGENTS.md - MemoFS

This is an open-source software (OSS) project.

## Behavioral Rules

- **Do not** add new npm dependencies without evaluating if an existing package already covers the need
- **Do not** use `console.log` in production code — use structured logging or remove it
- **Do not** commit secrets, API keys, or environment values — use `.env` files that are gitignored
- **Do not** run `pnpm build` during a code-editing session unless you are explicitly validating production correctness
- **Do not** add `prettier` — it has been removed; all formatting goes through Biome
- **Do not** use `@repo/` for public OSS packages — that scope is for internal tooling only
- **Do not** copy-paste tsdown options into new packages — import `pkgConfig` from `@repo/tsdown` instead
- **DRY & SSOT everywhere**: Enforce Single Source of Truth and Don't-Repeat-Yourself across the **entire workspace**, including the cloud app. Do not duplicate knowledge, logic, constants, or copy that already lives elsewhere — extract to a shared module, type, or constant and import it.
- **Always use** the `technical-writer` skill when writing package READMEs and user facing documentation.

## Cloud App Rules

- **Shadcn UI**: Always use the `shadcn` skill and use shadcn UI components if they exist for the use case.
- **File Naming**: Use **kebab-case** for all source files — `hero-section.tsx`, `side-bar.tsx`, `pricing-card.tsx`. Never PascalCase (`Hero.tsx`) or camelCase (`sideBar.tsx`). This applies to every `.ts`/`.tsx` file: route modules, `+components/`, `+utils/`, shared components, and pages. Vendored shadcn primitives in `components/ui/` keep their upstream casing (e.g. `dropdown-menu.tsx`) — they are generated, not hand-authored.
- **Component Length**: A single React component should be **80 lines**; **100 lines is a hard cap**. If it grows beyond that, extract sub-components into the route's `+components/` directory (colocated via `react-router-auto-routes` `+` convention).
- **File Length**: A file may contain multiple React components, but **MUST NOT exceed 500 lines** total. When a file approaches the limit, split it into colocated modules. Vendored shadcn primitives (`components/ui/*`, generated by `shadcn add`) are **exempt** — do not split them to satisfy the cap.
- **State Management**:
  - **Related State**: If you have up to 3 related states, use a single `useState` with an object.
  - **Unrelated State**: If you have up to 4 unrelated states, use `useReducer` to handle transitions.
- **`useEffect`**: Use the `useEffect` hook **only where necessary** — external synchronization, subscriptions, or browser APIs. Prefer loaders for data fetching and derived state for transformations. Do not reach for `useEffect` to derive values or orchestrate client-side data fetching.
- **Icons**: Always use **Lucide** icons via `lucide-react`. Do not introduce other icon libraries, use svgs if required.
- **ClassName Styling**: Always use `cn()` from `~/lib/utils` for merging Tailwind classes in apps/cloud:
  - Import : `import { cn } from "~/lib/utils"`
  - Pattern: `cn("default-style-1", "default-style-2", { "conditional-style": condition })`
  - For long default styles, break into multiple strings:
    ```tsx
    cn(
      "base-style-1",
      "base-style-2",
      "base-style-3",
      { "override": condition }
    )
    ```
  - Default/base styles go in string arguments (always applied).
  - Conditional/override styles go in the object argument (only applied when condition is true).
  - Never use template literals (`` `base ${condition && 'override'}` ``) for className.

## Route Module Conventions (Complex Routes Only)

This project uses **flat routing** via `react-router-auto-routes`. Simple routes (pure render components with no server logic) remain as single `*.tsx` files at the route path. The `routes.ts` file configures `colocationChar: "+"` for future use.

For **complex routes only** (routes with actions, loaders, multiple colocated files, or significant server-side logic), convert the flat file into a folder within `src/routes/` following these rules:

1. **Folder Name**: Defines the URL path (e.g., `src/routes/about/` -> `/about`). Use `$` for dynamic segments (e.g., `src/routes/projects/$projectId/` -> `/projects/:projectId`).
2. **Main Component**: Must be in `index.tsx` within the route folder (default export).
3. **Colocation**: All route-specific logic must be colocated in the same folder using these naming patterns:
   - `index.tsx`: The main route component.
   - `_layout.tsx`: Layout wrapper for the route and its children.
   - `+action.server.ts`: Route action (server-side logic).
   - `+loader.server.ts`: Route loader (server-side logic).
   - `+utils.ts`: Internal utilities scoped strictly to the route.
   - `+components/`: Components used by the route.
4. **Extraction Rule**: If an `action` or `loader` exceeds **5 lines of code**, it MUST be extracted from `index.tsx` and placed into `+action.server.ts` or `+loader.server.ts` respectively.
5. **Types**: Use React Router's `Route.LoaderArgs` / `Route.ActionArgs` types from `./+types/index.ts`.

**Example structure for `/projects/:id`**:

```
src/routes/projects/$id/
├── index.tsx            # Default component export
├── _layout.tsx          # Optional layout
├── +action.server.ts    # Optional action export
├── +client-action.ts    # Optional client action export
├── +loader.server.ts    # Optional loader export
├── +client-loader.ts    # Optional client loader export
├── +utils.ts            # Optional shared utilities
└── +components/         # Optional components
```

## Forms, Validation, and Submissions

All forms that cause page redirects, reloads, validate user input, or submit mutations MUST use Conform with Zod v4. Treat this as the default pattern for auth, contact, billing, onboarding, settings, and dashboard mutation flows.

- **Zod imports**: Always import Zod from `zod/v4`.
  ```ts
  import { z } from "zod/v4";
  ```
- **Conform Zod adapter**: Always import `parseWithZod` from `@conform-to/zod/v4`.
  ```ts
  import { parseWithZod } from "@conform-to/zod/v4";
  ```
- **Client form wiring**: Use `useForm`, `getFormProps`, `getTextareaProps`, and `getInputProps` from `@conform-to/react` for form state, form props, field props, default values, and validation errors.
- **Server submission parsing**: In route actions, read `request.formData()` and validate it with `parseWithZod(formData, { schema })`. Do not hand-roll form parsing or validation for structured form submissions.
- **Shared schema**: Define a route-local schema near the form or in the route's `+utils.ts` when reused by `index.tsx` and `+action.server.ts`. The same schema should power client `onValidate` and server action validation.
- **Async validation**: Use Zod v4 `superRefine` with `parseWithZod(..., { async: true })` for checks that require the database or external services, such as duplicate email validation.
- **Action responses**: If `submission.status !== "success"`, return `submission.reply()` with an appropriate HTTP status. Preserve field-level and form-level errors from Conform so the UI can render them without custom error-shape adapters.
- **Field errors**: Render Conform field errors next to the relevant input and form-level errors near the submit action. Prefer shared error UI components when available.
- **Pending UI**: Use `useDelayedIsPending` from `@repo/hooks/use-is-pending` or the fetcher pending state helper already present in the app when disabling submit buttons or showing loading indicators.
- **Route extraction**: If a form action exceeds 5 lines, keep the action in `+action.server.ts` according to the route module convention above.
